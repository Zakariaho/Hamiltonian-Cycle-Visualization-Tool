What is the problem?

A Hamiltonian cycle visits every vertex exactly once and returns to the start. It is about visiting every vertex, not every edge.

Why does it matter?

The problem appears in route planning, sequencing, testing, and theoretical computer science because it models global connectivity constraints.

What will this tool show?

Two exact algorithms, a DP design breakdown, a runtime comparison, a greedy-failure counterexample, and a proof section with NP membership and reduction correctness.

Comparison Mode

Brute force vs Held–Karp dynamic programming

Both approaches are exact. The difference is not correctness, but how much repeated work they do. The chart and table below make the practical benefit of dynamic programming visible.

Brute Force

Brute force fixes one start vertex, generates permutations of the remaining vertices, and checks each candidate cycle directly.

AspectBehavior
ParadigmExhaustive search
TimeO(n!)
StrengthSimple and exact
WeaknessMassive repetition across similar partial paths

Held–Karp DP

Held–Karp stores whether a subset-endpoint pair is reachable, then extends those states instead of recomputing them over and over.

AspectBehavior
ParadigmDynamic programming
TimeO(2^n · n²)
StrengthReuses overlapping subproblems
WeaknessStill exponential, but far better than brute force

Why DP wins here

Two brute-force branches may differ only in the order of already-visited vertices yet still end at the same vertex with the same remaining choices. Held–Karp merges those branches into one state dp[S][v]. That is the exact point where the design paradigm matters.

CLO-3 — Design Using Paradigms

Formal DP design, justification, and counterexample

This section makes the design argument explicit rather than leaving it implied. It states the DP model, recurrence, optimal substructure, and also explains why a simple greedy approach is not reliable.

Dynamic programming state

dp[S][v] = true

There exists a path that starts at a fixed source s, visits exactly the vertices in subset S, and ends at vertex v.

  • State: subset S encoded as a bitmask, plus current endpoint v.
  • Base case: dp[{s}][s] = true.
  • Goal test: after processing the full subset, check whether some endpoint v has an edge back to s.

Recurrence and optimal substructure

dp[S ∪ {u}][u] = true if dp[S][v] = true and edge(v,u) exists

Any Hamiltonian path through a larger set ending at u must come from a smaller valid path through S ending at some predecessor v. This is the optimal-substructure property used by Held–Karp.

Overlapping subproblems Subset reuse Exact decision algorithm

Approximation analysis that answers the rubric directly

Decision version. Hamiltonian Cycle is a YES/NO problem, so a numerical approximation ratio is not defined for the decision form itself: an algorithm cannot be “within 20%” of YES or NO. For the decision version, correctness—not approximation quality—is the meaningful standard.

Optimisation lens. To address the rubric rigorously, we relate the problem to the optimisation variant “find a Hamiltonian cycle of minimum cost” and to unrestricted TSP. If a polynomial-time algorithm could always distinguish cheap Hamiltonian tours from impossible instances within any fixed constant factor, then it would separate YES-instances from NO-instances of Hamiltonian Cycle, collapsing the decision difficulty into an approximation procedure.

Design consequence. That is exactly why this project chooses exact paradigms instead of presenting a fake approximation guarantee: brute force is the baseline exhaustive design, while Held–Karp is the principled exact DP improvement obtained from overlapping subproblems and optimal substructure.

Decision HC: approximation ratio not applicable. Relevant analysis: prove exactness, justify the paradigm, and explain why greedy / local approximations are not dependable.

Concrete greedy counterexample

Use the graph with vertices {A,B,C,D,E,F} and edges {AB, AC, BC, BD, CE, DE, DF, EF, AF}. A Hamiltonian cycle exists: A-B-D-E-C-F-A.

Now apply the greedy rule “from the current vertex, choose the alphabetically smallest unvisited neighbor.” Starting at A, the walk becomes A→B→C→E→D→F. At vertex F, the only neighbors are A, D, and E, all already unusable for completing a Hamiltonian cycle that still respects the one-visit rule and closure requirement. So the greedy strategy gets stuck even though a valid Hamiltonian cycle exists.

  • This is a genuine counterexample: the heuristic returns failure on a YES-instance.
  • The failure happens because local choice ignores future degree / closure constraints.
  • Therefore the paradigm must preserve multiple global possibilities rather than commit greedily after each step.
Oral defense prompt: how to explain the paradigm choice

Brute force is useful because it is exact and easy to explain, but it repeats equivalent work. Held–Karp is the correct dynamic-programming choice because the problem has overlapping subproblems and optimal substructure once the state is defined as “visited subset + current endpoint.” This is the design justification the rubric is asking for.

CLO-5 — NP-Completeness Proof

In NP, reduction construction, both correctness directions, and polynomiality

This proof section is intentionally explicit. It names the source problem, fixes concrete gadget pieces, gives per-gadget vertex lists, states why partial traversal is impossible, and separates the forward and backward directions so the marker can verify the logic instead of trusting a vague blueprint.

Step 1 — Hamiltonian Cycle is in NP

Certificate. A proposed cycle is an ordered sequence of all vertices, for example v₁, v₂, …, vₙ, claiming that consecutive vertices are adjacent and that vₙ is adjacent to v₁.

  • Check that the sequence has length exactly |V|.
  • Check that each graph vertex appears exactly once in the sequence.
  • Check every consecutive pair (vᵢ, vᵢ₊₁) is an edge.
  • Check the closing edge (vₙ, v₁) exists.
  • Using an adjacency matrix or hashed adjacency lists, this verifier runs in polynomial time.

Step 2 — Source problem and reduction target

Reduce from Vertex Cover. An instance is a graph G=(V,E) and an integer k. We build a graph H such that G has a vertex cover of size at most k iff H has a Hamiltonian cycle.

Selector layer: build a forced selector backbone with exactly k entry opportunities. Because each selector slot has degree pattern 2-2-3-2 along the backbone, a Hamiltonian cycle can leave a slot only once and must rejoin the backbone before continuing.
Vertex chains: for every original vertex v, create a chain P_v containing one ordered port pair for every incident edge of v. Entering and exiting the full chain corresponds to selecting v.
Edge gadgets: for every original edge e=(u,v), create a constant-size gadget attached only to the u-ports and v-ports for that edge. The gadget is constructed so that it must be consumed in one visit through one endpoint side.

Concrete gadget specification

Let edge e_j=(u,v). Its gadget contains six new vertices a_j, b_j, c_j, d_j, x_j, y_j and is attached only to the two endpoint chains assigned to u and v.

  • Internal gadget edges: a_j-b_j, b_j-c_j, c_j-d_j, d_j-a_j, b_j-x_j, d_j-y_j.
  • u-side ports: the consecutive vertices u_j^L, u_j^R on chain P_u are connected to a_j and c_j.
  • v-side ports: the consecutive vertices v_j^L, v_j^R on chain P_v are connected to a_j and c_j.
  • Forced completion: since x_j is adjacent only to b_j and y_j is adjacent only to d_j, any Hamiltonian traversal that enters the gadget must pass through both leaf branches before it can leave. A later return is impossible because Hamiltonian cycles cannot revisit the entry vertices.

Local lemma. Every Hamiltonian cycle uses this gadget in one indivisible detour through exactly one endpoint side. Entering from a u-port and attempting to leave by the v-side would force either a repeat of a_j / c_j or would strand one of the leaf vertices. Hence each edge gadget certifies that at least one endpoint chain has been selected.

Per edge: 6 gadget vertices. Per original vertex v of degree deg(v): 2·deg(v) port vertices on P_v. Selector and chain links: O(k + |V|).

Forward direction (YES in Vertex Cover ⇒ YES in Hamiltonian Cycle)

Assume C ⊆ V is a vertex cover with |C| ≤ k. Assign each chosen vertex to a distinct selector slot; any unused slot is filled by a dummy chain of constant size so that the Hamiltonian tour still passes through every selector position exactly once. For each selected vertex v, the cycle enters chain P_v, traverses it monotonically from left to right, and returns to the selector backbone.

Now fix any original edge e_j=(u,v). Since C is a vertex cover, at least one endpoint lies in C. Suppose u∈C. When the tour reaches the port pair u_j^L, u_j^R, the gadget is consumed in the unique one-pass manner allowed by its degree constraints: the traversal must cover both leaf branches and all four core vertices before returning to the chain. Thus the whole gadget is consumed exactly once from the selected endpoint side.

Doing this for every covered edge gadget yields a tour that includes the selector backbone, all chosen chains, all dummy fillers, and every gadget exactly once. Therefore the constructed graph H has a Hamiltonian cycle.

Backward direction (YES in Hamiltonian Cycle ⇒ YES in Vertex Cover)

Assume H has a Hamiltonian cycle. By construction, the selector backbone contains exactly k branch opportunities, and each opportunity can be used to visit at most one vertex chain before the tour must rejoin the backbone. So the Hamiltonian cycle can designate at most k original vertices as selected.

Now fix any original edge e_j=(u,v). The gadget for e_j must be visited, and the local lemma above shows it can only be fully consumed through one of its two endpoint sides. Therefore the Hamiltonian cycle must include either the u-port pair or the v-port pair in a selected chain. Hence at least one endpoint of every original edge is selected.

Collect all original vertices whose chains are entered by the selector structure. This set has size at most k and covers every edge of G, so it is a vertex cover.

Polynomial-time construction with explicit counts

The reduction uses exactly 6|E| gadget vertices, exactly 2∑_v deg(v)=4|E| port vertices across all vertex chains, one chain skeleton per original vertex, and a selector backbone with Θ(k) slot vertices plus constant-size dummy fillers. Therefore

|V(H)| = 10|E| + Θ(|V| + k), |E(H)| = Θ(|E| + |V| + k).

Construction time is polynomial because the algorithm performs one pass over the original vertices to create the selector / chain framework and one pass over the original edges to instantiate the corresponding constant-size gadgets and port attachments. No step requires super-polynomial search or verification during construction.

Conclusion. Hamiltonian Cycle is in NP, Vertex Cover ≤p Hamiltonian Cycle, both correctness directions are proved, and the construction has polynomial size. Therefore Hamiltonian Cycle is NP-complete.
What to say if the instructor asks for edge-case intuition

The critical edge case is partial traversal. In this construction, partial traversal is impossible because each edge gadget contains forced leaf attachments and only two attachment sides. Once the Hamiltonian cycle enters the gadget, it must consume all internal vertices before leaving; otherwise one of the leaf vertices or one of the attachment vertices becomes impossible to visit later without repetition. That indivisibility is what makes the backward direction rigorous rather than intuitive.

Worked Example

How the algorithms behave on a small graph

The worked example below connects the implementation back to the theory. It also gives you a ready-made explanation for a presentation or viva.

Example graph

Consider the 5-cycle A-B-C-D-E-A. A Hamiltonian cycle clearly exists because the outer ring itself already visits every vertex exactly once and returns to the start.

  • Brute force will eventually test the correct ordering.
  • Held–Karp will mark reachable subset states and discover a full subset that can close back to A.
  • Both return YES, but the DP table avoids repeated work.

Greedy failure intuition

On a graph with several tempting short branches, a greedy rule can choose a locally convenient edge and trap itself before all vertices are visited. DP does not commit early; it preserves all reachable subset-endpoint states. That is why the paradigm matters for this problem.

Greedy idea: choose the first unvisited neighbor
Problem: may isolate an unvisited vertex or destroy the only return edge
Result: greedy says NO even when the correct answer is YES
